


초장기(100라운드 이상) 멀티 에이전트 토론 환경에서 **비용(토큰)은 고정하면서 완벽한 기억력을 유지**하기 위해, 앞서 논의한 **[JSON 상태 추적 + BM25 기반 RAG + 단기 롤링 윈도우]**를 결합한 시스템 기획서를 작성했습니다.

기존 `Ship Council`의 기술 스택(Node.js, SQLite)과 완벽하게 호환되며, 무거운 벡터 DB(Vector DB)나 임베딩 API 호출 비용 없이 **SQLite FTS5(Full-Text Search)와 BM25 알고리즘**을 활용해 가볍고 빠르게 구현하는 방향으로 기획되었습니다.

# [프로젝트 기획서] 무한 컨텍스트 멀티 에이전트 브레인스토밍 시스템 (가칭: Infinity Council)

## 1. 프로젝트 개요 및 배경
현재의 LLM 기반 멀티 에이전트 토론 시스템은 대화가 길어질수록(10~20회 이상) 과거의 대화 기록을 단순히 이어 붙이는 방식을 사용하여 심각한 문제에 직면합니다.
1. **컨텍스트 윈도우 초과 (Context Limit):** 토론이 중단되거나 시스템 에러 발생.
2. **비용의 기하급수적 증가 (Token Cost):** 매 턴마다 누적된 수만 개의 토큰을 재전송하여 막대한 API 비용 발생.
3. **망각 현상 (Lost in the Middle):** 요약본이 길어지면서 초반에 논의된 중요한 디테일이나 아이디어가 소실됨.

**목표:** 100번~1,000번의 브레인스토밍 토론을 거쳐도 **입력 토큰 크기를 항상 일정하게(O(1)) 유지**하고, **과거의 중요한 발언 디테일을 절대 잊지 않는** 3단계 하이브리드 메모리 아키텍처를 구축합니다.

---

## 2. 시스템 핵심 아키텍처: 3-Tier 하이브리드 메모리

토론에 참여하는 에이전트(LLM)에게 전체 대화 기록을 주지 않고, 인간의 뇌가 기억을 처리하는 방식과 동일한 3단계 기억 시스템을 프롬프트로 구성하여 제공합니다.

### 🧠 Tier 1: 단기 기억 (Rolling Window)
* **목적:** 자연스러운 대화의 흐름과 직전 맥락 유지
* **방식:** 가장 최근에 발생한 N개(예: 5~8개)의 원본 발언(Raw Text)만 프롬프트에 유지.
* **효과:** "방금 네가 한 말에 반박하자면~" 과 같은 즉각적인 상호작용 보장.

### 📊 Tier 2: 작업 기억 (JSON State Tracking / 화이트보드)
* **목적:** 토론의 현재 진행 상황과 논리적 뼈대 유지 (재귀적 덮어쓰기)
* **방식:** 매 라운드(또는 N턴)마다 백그라운드 LLM이 개입하여 토론 상태를 JSON 형태로 **'업데이트(Overwrite)'** 합니다. 
* **데이터 구조 예시:**
  ```json
  {
    "agreed_points":["MVP는 웹 기반으로 출시", "결제 연동은 Stripe 사용"],
    "active_conflicts":["A(디자이너): 커스텀 UI 필요 vs B(엔지니어): 초기엔 템플릿 사용"],
    "pending_questions":["초기 마케팅 채널은 무엇으로 할 것인가?"]
  }
  ```
* **효과:** 100번을 토론해도 요약본의 길이는 고정되며, 에이전트들이 현재 무엇을 합의했고 무엇을 싸우고 있는지 명확히 인지하여 중복 발언(Hallucination)을 방지합니다.

### 🔍 Tier 3: 장기 기억 (BM25 기반 RAG - 영구 기억소)
* **목적:** 요약 과정에서 소실되는 과거의 '디테일(근거, 예시, 과거 발언)'을 정확히 소환.
* **방식:** 에이전트의 모든 발언을 DB에 저장하고, 현재 발언할 차례인 에이전트의 역할과 직전 대화의 '키워드'를 기반으로 **BM25 (어휘 기반 검색 알고리즘)** 를 사용해 가장 연관성 높은 과거 발언 Top 3를 검색하여 프롬프트에 주입.
* **왜 임베딩(Vector DB)이 아닌 BM25인가?**
  * **비용 Zero:** 임베딩 모델(OpenAI `text-embedding-3` 등)을 호출할 필요가 없어 토큰 비용이 0원입니다.
  * **정확한 키워드 매칭:** "아까 10라운드에서 말했던 보안(Security) 이슈"를 찾을 때, 의미론적 유사도(Vector)보다 정확한 단어 매칭(BM25)이 토론의 근거를 가져오는 데 훨씬 유리합니다.
  * **시스템 경량화:** 현재 `Ship Council`이 사용하는 `better-sqlite3`의 **FTS5 (Full-Text Search)** 모듈을 활용하면 외부 DB 구축 없이 내장 기능만으로 고성능 BM25 검색을 즉시 구현할 수 있습니다.

## 3. BM25 RAG 통합 토론 플로우 (Execution Workflow)

에이전트 '엔지니어(Engineer)'가 51번째 턴에서 발언해야 할 때의 시스템 동작 플로우입니다.

1. **키워드 추출 (Keyword Extraction):**
   * 현재 직전 대화(최근 2~3개)에서 주요 명사/키워드 추출 (예: "결제", "트랜잭션", "보안").
2. **BM25 검색 (Memory Retrieval):**
   * SQLite FTS5를 통해 과거 50번의 대화 기록 중 해당 키워드와 가장 연관성 높은 과거 발언(예: 12턴 디자이너 발언, 25턴 회의론자 발언) Top 3를 검색.
3. **프롬프트 조립 (Prompt Assembly):**
   * LLM에게 전달할 최종 프롬프트를 아래와 같이 조립합니다.
     ```text[시스템 프롬프트 & 에이전트 페르소나][현재 토론 화이트보드 (JSON State)]
     - 합의된 사항: ...
     - 대립 중인 사항: ...
     
     [💡 과거 기억 소환 (BM25 Retrieved Memory)]
     - 12턴 (디자이너): "결제 시 UX 뎁스를 줄여야 합니다."
     - 25턴 (회의론자): "결제 실패 시 트랜잭션 롤백 처리가 불명확합니다."[최근 대화 5개 (Rolling Window)]
     - 49턴 (PM): "..."
     - 50턴 (디자이너): "..."
     
     당신의 차례입니다. 논리를 전개하세요.
     ```
4. **발언 생성 및 DB 저장:**
   * 에이전트가 발언을 생성하면, 이 텍스트는 즉시 SQLite FTS5 가상 테이블에 색인(Index)되어 다음 검색의 대상이 됩니다.
5. **토론 상태 업데이트 (Background):**
   * N턴 주기로 백그라운드에서 요약 프롬프트가 실행되어 [작업 기억 JSON]을 갱신합니다.

## 4. 기존 대비 기대 효과 (ROI)

| 구분 | 기존 방식 (`join` 기반 요약) | **Infinity Council (JSON + BM25)** |
| :--- | :--- | :--- |
| **최대 토론 횟수** | 10~20회 (이후 컨텍스트 초과 위험) | **무제한 (100회, 1000회 가능)** |
| **토큰 발생 비용** | $O(N^2)$ - 턴마다 이전 텍스트 누적 전송 | **$O(1)$ - 매 턴 고정된 토큰 크기 전송** |
| **디테일 유지력** | 낮음 (요약될수록 구체적 수치, 예시 증발) | **매우 높음 (BM25로 원본 발언 직접 인용)** |
| **인프라 복잡도** | 낮음 (단순 텍스트 결합) | 보통 (SQLite 내장 FTS5 사용으로 외부 DB 불필요) |

## 5. 단계별 개발 마일스톤 (Action Plan)

* **Phase 1: 기반 구조 변경 (State Tracking)**
  * 기존 `compressedSummary` 문자열 필드를 제거하고, `debate_state` (JSON) 구조로 변경.
  * 오케스트레이션(`src/run-session.ts`)에서 주기적으로 JSON 상태를 덮어쓰는(Overwrite) 프롬프트 로직 구현.
* **Phase 2: BM25 / FTS5 통합 (Memory DB)**
  * `packages/shared/src/db.ts`에 SQLite FTS5 가상 테이블(`messages_fts`) 추가.
  * 메시지가 저장될 때마다 FTS 테이블에 자동 동기화되도록 트리거(Trigger) 작성.
* **Phase 3: RAG 파이프라인 연동 (Retrieval)**
  * 에이전트 발언 생성 직전에 최적의 검색어(Search Query)를 도출하여 FTS5 쿼리 실행.
  * 검색된 텍스트를 `formatOpinionPrompt`, `formatRebuttalPrompt`의 컨텍스트 섹션에 주입하는 로직 구현.
* **Phase 4: UI/UX 고도화**
  * 웹 UI에 "토론 화이트보드(현재 JSON 상태)"를 실시간 패널로 노출하여 사용자가 토론의 맥락을 한눈에 볼 수 있도록 시각화.
  * AI가 과거 발언을 참고했을 경우 "💡 12라운드 회의론자의 발언 인용" 과 같은 출처(Citation) 표시.
