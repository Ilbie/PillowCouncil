



**PillowCouncil**을 누구나 터미널에서 `npx pillow-council` 또는 `npm install -g pillow-council` 명령어 한 줄로 간편하게 설치하고 실행할 수 있도록 변환하기 위한 **[NPM CLI 배포용 아키텍처 및 코드 수정 기획서]**를 작성해 드립니다.

이 기획서는 현재 로컬 개발 환경(`npm run dev`)에 강하게 결합된 코드베이스를 **독립 실행형(Standalone) 글로벌 패키지**로 탈바꿈시키는 데 목적이 있습니다.

# 🚀 PillowCouncil NPM CLI 배포 수정 기획서

## 🎯 1. 개요 및 목표
*   **목표:** Windows, Mac, Linux 환경에서 복잡한 환경 설정(env, 소스코드 클론 등) 없이 NPM 명령어로 즉시 PillowCouncil 로컬 서버를 구동하고 브라우저를 띄움.
*   **핵심 과제:**
    1.  NPM 패키지 업데이트 시 사용자의 기존 토론 내역과 설정(DB)이 날아가지 않도록 **데이터 영속성 보장**.
    2.  어떤 폴더에서 실행하든 내부 엔진(OpenCode)을 정상적으로 찾아 실행할 수 있도록 **경로 동적 탐색**.
    3.  소스코드 없이도 실행 가능한 **Next.js Standalone 빌드** 및 **CLI 진입점(Entrypoint)** 구축.

---

## 🛠️ 2. 단계별 수정 기획 (Action Plan)

### Phase 1. 데이터 및 설정 영속성 보장 (사용자 홈 디렉토리 마이그레이션)
현재 코드는 프로젝트 폴더(`workspaceRoot`) 내에 DB와 OpenCode 설정을 저장합니다. NPM 글로벌 설치 환경에서는 패키지가 업데이트될 때 해당 폴더가 삭제/덮어쓰기 되므로, **모든 데이터를 OS의 사용자 홈 디렉토리(Home Directory) 하위 숨김 폴더로 이동**해야 합니다.

*   **수정 파일 1:** `packages/shared/src/utils.ts`
    *   **수정 내용:** `databasePath()` 함수가 반환하는 경로를 `os.homedir() + '/.pillow-council/data/pillow-council.db'`로 변경.
*   **수정 파일 2:** `packages/providers/src/opencode.ts`
    *   **수정 내용:** `getOpencodeDirectory()` 함수가 `workspaceRoot()` 대신 `os.homedir() + '/.pillow-council'`을 반환하도록 수정.
*   **수정 파일 3:** `packages/providers/src/settings-runtime.ts`
    *   **수정 내용:** `ENABLED_SKILLS_DIR` 및 `DISABLED_SKILLS_DIR` 상수의 기준 경로를 `os.homedir() + '/.pillow-council/skills'` 등으로 변경하여 스킬 데이터 보존.

### Phase 2. 내부 의존성(OpenCode) 바이너리 동적 탐색
NPM 글로벌 패키지로 설치하면 `node_modules`의 위치 구조가 달라집니다. 따라서 하드코딩된 폴더 탐색 로직은 런타임 에러를 발생시킵니다.

*   **수정 파일:** `packages/providers/src/opencode.ts`
    *   **수정 내용:** `getOpencodeLauncherPath()` 함수 내에서 `workspaceRoot()` 기반 탐색을 제거하고, Node.js의 모듈 해석기인 **`require.resolve('opencode-ai/package.json')`**를 사용하여 `opencode-ai` 패키지의 설치 위치를 동적으로 알아낸 뒤, 해당 경로 안의 `bin/opencode`를 실행하도록 수정.

### Phase 3. 독립 실행형(Standalone) 서버 빌드
NPM 패키지 용량을 최적화하고, `npm run dev` 없이 실행되게 하려면 Next.js의 프로덕션 빌드 결과물만 패키징해야 합니다.

*   **수정 파일:** `apps/web/next.config.ts`
    *   **수정 내용:** `output: "standalone"` 속성 추가.
    *   **기대 효과:** 빌드 시 `.next/standalone` 폴더에 `node_modules`와 서버 구동에 필요한 최소한의 파일이 모여, 소스코드 없이 `node server.js`만으로 서버 구동이 가능해짐.

### Phase 4. CLI 실행 스크립트 구축 및 자동화
사용자가 터미널에 `pillow-council`을 입력했을 때, 서버를 켜고 브라우저를 자동으로 띄워주는 "실행 껍데기"를 만듭니다.

*   **신규 파일:** `apps/web/bin/cli.js` (또는 프로젝트 루트 `bin/cli.js`)
    *   **구현 로직:**
        1.  `#!/usr/bin/env node` 셰뱅(Shebang) 선언.
        2.  빈 포트(예: 3000)를 탐색하고 환경변수(`PORT=3000`, `NODE_ENV=production`) 설정.
        3.  `child_process.spawn`을 사용해 `.next/standalone/apps/web/server.js` 백그라운드 실행.
        4.  서버 구동 대기 후(약 1~2초), `open` 라이브러리를 사용하여 사용자의 기본 웹 브라우저(`http://localhost:3000`) 자동 실행.
*   **수정 파일:** 루트 `package.json`
    *   **수정 내용:** `"bin": { "pillow-council": "bin/cli.js" }` 항목 추가.

### Phase 5. 패키지 발행(Publish) 설정
모노레포 구조이므로 전체 소스코드가 아닌, 빌드된 결과물만 NPM에 업로드되도록 필터링합니다.

*   **신규 파일:** 루트 `.npmignore`
    *   **구현 내용:** `src/`, `tests/`, `.github/`, 모노레포 원본 패키지 등을 제외하고, 오직 `bin/` 폴더와 `apps/web/.next/standalone/` 폴더만 NPM에 업로드되도록 설정.

## 🚀 3. 기대되는 배포 및 사용 시나리오

이 기획서대로 수정을 완료하고 `npm publish`를 통해 배포하게 되면, 사용자는 다음과 같은 경험을 하게 됩니다.

**1. 간편한 최초 실행 (Zero-Configuration)**
```bash
# 설치와 동시에 바로 실행 (권장)
npx pillow-council
```
*   명령어 입력 즉시 브라우저가 열리며 `http://localhost:3000`에 접속됨.
*   최초 실행 시 `~/.pillow-council/data/pillow-council.db`가 자동 생성되어 세팅 시작.

**2. 데이터가 보존되는 안전한 업데이트**
```bash
# 글로벌 설치자의 경우 업데이트
npm update -g pillow-council
```
*   NPM 패키지가 업데이트되어도 `~/.pillow-council` 폴더는 건드리지 않으므로, 사용자의 기존 연결 설정, 저장된 커스텀 프리셋, 지난 토론 내역이 100% 안전하게 유지됨.
