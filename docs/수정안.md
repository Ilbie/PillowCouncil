제공해주신 스크린샷의 기괴한 결과물(경제 논의나 아이디어 기획이 시스템 아키텍처/에러 처리 명세서로 변질되는 현상)은 단순히 AI 모델의 일시적인 오류가 아닙니다. 

이는 **Ship Council 코드베이스의 오케스트레이션(토론 진행) 로직과 시스템 프롬프트가 결합하여 만들어낸 구조적인 문제(Feedback Loop)**입니다.

코드와 프롬프트 레벨에서 정확히 어떤 부분이 원인인지 심층 분석하고, 이를 근본적으로 해결하기 위한 **코드 수정안(Modification Plan)**을 기획해 드립니다.

## 🔍 1. 심층 원인 분석 (코드 & 프롬프트 레벨)

### 원인 A. `Debate State` (칠판) 요약 로직의 '주제 이탈(Topic Hijacking)' 현상
**위치:** `packages/orchestration/src/run-session.ts` 내 `refreshDebateState` 함수

이 앱은 여러 번의 라운드를 거치면서 토론이 길어질 때 컨텍스트 제한을 막기 위해, 매 라운드마다 토론 내용을 요약하여 `DebateState`(합의점, 갈등, 남은 질문) JSON으로 압축합니다. 

*   **문제점:** `refreshDebateState`의 시스템 프롬프트에는 **"원래의 주제(Original Topic)를 잃지 마라"는 방어 로직이 부족**합니다. 
*   만약 첫 라운드에서 엔지니어 에이전트가 "이 아이디어는 에러 처리가 안 되어 있다"라고 지엽적인 태클을 걸면, 요약 모델은 이를 `activeConflicts`(활성 갈등)에 기록해버립니다. 
*   다음 라운드의 에이전트들은 원래 주제("SaaS 아이디어")는 잊어버리고, 칠판에 적힌 '에러 처리' 갈등을 해결하는 데에만 모델의 연산력을 쏟아붓게 됩니다.

### 원인 B. 공격성만 강요하는 `Rebuttal`(반박) 프롬프트
**위치:** `packages/orchestration/src/prompts.ts` 내 `formatRebuttalPrompt` 함수

```typescript
// 현재 반박 프롬프트 중 일부
"Select exactly one target agent for this rebuttal.",
"Attack contradictions, weak assumptions, missing evidence, and execution risk."
```
*   **문제점:** 모델에게 억지로 **"모순, 약한 가정, 누락된 증거, 실행 리스크를 공격(Attack)하라"**고 강제하고 있습니다. 
*   브레인스토밍(SaaS 아이디어)이나 거시 경제(IMF)를 논할 때 이런 프롬프트가 주어지면, AI는 억지로 꼬투리를 잡기 위해 가장 만만한 **'기술적 세부 구현'이나 '극단적 예외 상황(Edge-case)'을 파고듭니다.** 이로 인해 논의가 마이크로 매니징과 시스템 로직으로 빠지게 됩니다.

### 원인 C. 에이전트 페르소나의 '과적합(Overfitting)'과 도메인 미스매치
**위치:** `packages/agents/src/presets.ts`

```typescript
// 현재 Staff Engineer 시스템 프롬프트
"구조, 실패 조건, 운영 비용을 짧고 정확하게 말한다... 장애 지점, 기술 부채를 냉정하게 검토한다."
```
*   **문제점:** 각 에이전트의 역할이 '소프트웨어 개발'에 너무 과적합되어 있습니다. 경제(IMF) 주제를 다루더라도 Staff Engineer는 무조건 '장애 지점'과 '시스템 아키텍처' 관점에서 대답해야 한다는 지시를 받습니다. 그 결과 거시경제 정책을 '2중 게이트, 72시간 경보 라우팅' 같은 서버 개발 용어로 뱉어내게 됩니다.

## 🛠️ 2. 코드 및 프롬프트 수정안 기획 (Action Plan)

이러한 악순환을 끊고 어떤 주제든 자연스럽고 본질에 집중하는 토론을 만들기 위해 다음 3가지 영역의 코드를 수정해야 합니다.

### 수정안 1. 토론 상태(Debate State) 압축 시 '주제 수호(Guardrail)' 로직 추가
토론이 지엽적인 기술/예외 처리로 빠지는 것을 막기 위해 `refreshDebateState` 프롬프트를 수정해야 합니다.

**파일:** `packages/orchestration/src/run-session.ts`
**수정 대상:** `refreshDebateState` 함수의 `system` 프롬프트 배열

```typescript
// [기존 코드]
system: [
  "You maintain a compact JSON whiteboard for a long-running debate.",
  "Overwrite the state using only concrete agreements, active conflicts, and unanswered questions.",
  // ...
]

// [수정 제안]
system: [
  "You maintain a compact JSON whiteboard for a long-running debate.",
  "Overwrite the state using only concrete agreements, active conflicts, and unanswered questions.",
  "CRITICAL INSTRUCTION: You MUST strictly align the whiteboard with the Original Topic. If the recent discussion has derailed into minor technical implementation details, error handling, or edge-case nitpicking that loses the big picture, IGNORE them and do not record them as active conflicts.", // <-- 핵심: 지엽적 기술/에러 논의 필터링 지시 추가
  "Avoid duplicates and keep each item short.",
  // ...
]
```

### 수정안 2. '반박(Rebuttal)' 프롬프트를 '건설적 비판'으로 연화
무조건적인 공격(Attack) 대신, 원래의 목적(주제 해결)에 맞는 건설적 반박을 유도하도록 프롬프트를 수정합니다.

**파일:** `packages/orchestration/src/prompts.ts`
**수정 대상:** `formatRebuttalPrompt` 함수

```typescript
// [기존 코드]
"Attack contradictions, weak assumptions, missing evidence, and execution risk.",

// [수정 제안]
"Provide a constructive critique of the target's claim.",
"If the target's claim is too abstract, demand concrete examples. If the target is bogged down in minor implementation details (like coding or error handling), redirect the focus back to the core Topic and business/user value.", // <-- 기술적 마이크로매니징을 멈추고 본질로 돌아오게 하는 지시
"Identify logical flaws or execution risks, but do not attack merely for the sake of attacking."
```

### 수정안 3. 에이전트 프롬프트에 '도메인 적응성(Adaptability)' 부여
기술적 에이전트(Engineer, Security 등)가 비기술적 주제(경제, 기획 등)를 다룰 때, 강박적으로 코딩 용어를 쓰지 않도록 유연성을 부여해야 합니다.

**파일:** `packages/orchestration/src/prompts.ts`
**수정 대상:** `formatAgentSystem` 함수

```typescript
// [기존 코드]
export function formatAgentSystem(agent: AgentDefinition, session: SessionRecord): string {
  return [
    agent.systemPrompt,
    `Role: ${agent.role}`,
    `Goal: ${agent.goal}`,
    // ...
  ].join("\n");
}

// [수정 제안]
export function formatAgentSystem(agent: AgentDefinition, session: SessionRecord): string {
  return [
    agent.systemPrompt,
    `Role: ${agent.role}`,
    `Goal: ${agent.goal}`,
    `Bias: ${agent.bias}`,
    `Style: ${agent.style}`,
    "IMPORTANT ROLE ADAPTATION: Adapt your role to the nature of the Topic. If the Topic is about business strategy, macro-economics, or high-level ideation, DO NOT use software engineering jargon, architecture specs, or code-level error handling logic. Express your role's core philosophy using general business or domain-appropriate language.", // <-- 주제의 성격에 맞춰 용어를 순화하라는 메타 프롬프트 추가
    // ...
  ].join("\n");
}
```

### 수정안 4. 최종 결론(Decision)의 스키마 유연화
현재 무조건 `risks`(위험 요소) 배열을 강제 생성하게 되어 있어, 아이디어 도출 같은 세션에서도 억지로 리스크를 만들어냅니다. 모데레이터가 주제에 집중하도록 최종 프롬프트를 다듬습니다.

## 💡 요약 및 적용 가이드

현재 Ship Council은 **"비판과 갈등을 찾아내라"**는 프롬프트 지시와 **"직전 라운드의 갈등을 칠판에 적어라"**는 시스템이 맞물려, 원래 주제를 버리고 **가장 꼬투리 잡기 쉬운 '기술적 예외 상황(Error/Risk)'으로 토론이 블랙홀처럼 빨려 들어가는 치명적인 논리적 결함**을 겪고 있습니다.

위에 제시해 드린 **4가지 프롬프트/오케스트레이션 코드 수정안**을 적용하시면:
1. 모데레이터가 토론 요약 시 지엽적인 시스템/에러 논의를 잘라냅니다.
2. 에이전트들이 상대방의 주장을 반박할 때 억지 기술적 꼬투리 잡기를 멈춥니다.
3. 엔지니어 역할이라도 경제나 기획 주제에서는 코딩 용어가 아닌 상식적인 언어로 토론에 참여하게 됩니다. 

이 코드를 적용하신 후 다시 실행해보시면 스크린샷과 같은 기괴한 '소프트웨어 명세서' 형태의 토론이 사라지고, 본래 목적에 맞는 비즈니스/거시적 토론이 이루어질 것입니다.
